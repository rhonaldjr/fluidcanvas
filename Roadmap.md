# InkPad Roadmap

Tasks are ordered and sized so each one is a single Claude Code session: "implement task X.Y". Each task includes its tests. Don't start a task until everything it depends on is checked off. Check tasks off here as they're completed.

InkPad is both a freehand sketching tool and a light diagramming tool: strokes and parametric shapes are peers. Both are `CanvasElement`s, live in the same ordered list inside a layer, and share one selection/transform system.

**InkPad is multi-document.** Several documents are open at once, each in its own tab. The tab *UI* is deliberately deferred to Phase 12, but the *state* is multi-document from task 2.5 onward, because that is the part that cannot be retrofitted cheaply. Everything belonging to one open document — the `SkdDocument`, its command stack, its selection, its viewport transform, its file path, its dirty flag — lives in a `DocumentSession` keyed by session id. Never reach for "the" document; always resolve it from the active session.

What is **per-session**: document, undo/redo history, selection, zoom/pan, file path, dirty flag. What stays **global**: the active tool, brush settings, and recent colors — switching tabs must not change which brush you are holding.

## Phase 1 — Open Source & Linux Distribution

InkPad ships as a public GitHub repository. For now the only supported platform is **Linux (AppImage, x86_64)**, developed and tested on Debian-based systems. macOS builds are deferred to Phase 16. The `windows/` target stays in the tree and builds locally, but no official Windows binaries are published.

This phase runs early on purpose: the release pipeline is easier to get right against a trivial app, and every later phase then lands with working CI and a proven path to a downloadable binary. Artifacts produced here are pre-alpha — they open an empty window and nothing more.

- [x] **1.1 Repository hygiene.** `LICENSE` (MIT), `CONTRIBUTING.md` (build steps, the analyze/test bar, conventional commits), `CODE_OF_CONDUCT.md`, GitHub issue and pull-request templates, and a `CHANGELOG.md` seeded with the first release.
- [x] **1.2 Linux AppImage.** Workflow job on the oldest supported Ubuntu runner (its glibc sets the floor for which distros can run the output). `flutter build linux --release` → assemble an AppDir with a `.desktop` file and the app icon at every hicolor size → package with `linuxdeploy` plus `linuxdeploy-plugin-gtk` (Flutter's embedder is a GTK app; the plugin bundles the pixbuf loaders and GIO modules plain linuxdeploy misses), then pack with `appimagetool` so the root `.DirIcon` is the 256px PNG rather than whichever size linuxdeploy picks → `InkPad-x86_64.AppImage`. Smoke-test the artifact in CI under `xvfb`: it launches and is still running ten seconds later. Task 13.6 later extends the `.desktop` file with the `.skd` MIME association.
- [x] **1.3 Linux release workflow.** On a `v*` tag: build the AppImage from 1.2, publish a GitHub Release, attach the AppImage, and generate a `SHA256SUMS` file. Release notes drawn from `CHANGELOG.md`. Mark pre-1.0 releases as pre-release. Phase 15 extends this workflow with the macOS artifacts.

## Phase 2 — Core Domain Models

- [x] **2.1 CanvasElement + Stroke models.** Sealed `CanvasElement {id}` with an abstract `bounds` getter. Immutable `StrokePoint {x, y, pressure}` and `Stroke extends CanvasElement {colorRGBA, baseWidth, toolId, points}` with `copyWith`, equality, and unit tests (including `bounds` of a zero- and one-point stroke).
- [x] **2.2 Shape model.** `ShapeType {rectangle, ellipse, line, arrow, diamond}`, `StrokeStyle {solid, dashed, dotted}`, and `Shape extends CanvasElement {type, x, y, w, h, rotation, strokeColorRGBA, fillColorRGBA, strokeWidth, strokeStyle}`. `fillColorRGBA` with alpha 0 means unfilled. Includes `normalized()` (folds negative `w`/`h` into `x`/`y`) and rotation-aware `bounds`. `copyWith`, equality, unit tests.
- [x] **2.3 Layer model.** Immutable `Layer {id, name, visible, opacity, blendMode, elements}` where `elements` is an ordered `List<CanvasElement>`, bottom to top. Add/remove/replace/reorder-element copy helpers. Unit tests.
- [x] **2.4 SkdDocument model.** `SkdDocument {canvasWidth, canvasHeight, backgroundRGBA, layers}` with helpers: `newDefault()` (one empty layer), layer lookup by id, replace-layer, and `findElement(id)` returning the owning layer plus the element. A document always holds at least one layer, and layer ids are unique. Unit tests.
- [x] **2.5 Session state provider.** `DocumentSession {id, document, activeLayerId}` — the per-open-document state bag that later phases extend with the command stack (6.1), selection (9.3), viewport (14.1), and file path + dirty flag (13.1, 13.3). A `SessionsNotifier` holds an ordered `sessions` list (tab order) plus `activeSessionId`, with `openSession(SkdDocument)`, `closeSession(id)`, and `setActiveSession(id)`. An `activeSessionProvider` resolves the current one; document mutations like `addElementToActiveLayer(CanvasElement)` act on it. Bootstrapping creates exactly one session from `SkdDocument.newDefault()`. Unit tests via `ProviderContainer`: two sessions stay independent, closing the active one selects a neighbour, closing the last one leaves a fresh empty session.

## Phase 3 — Draw a Line on Screen

- [x] **3.1 Pointer capture.** Canvas widget wraps the page in a `Listener`; on pointer down/move/up, collect raw `StrokePoint`s (use `event.pressure`, default 1.0 for mouse) converted into document space. For now, just `debugPrint` the point count on pointer-up.
- [x] **3.2 In-progress stroke rendering.** `CustomPainter` that renders a list of points as a stroked polyline with round caps/joins at fixed width and color. Live repaint while dragging via a `currentStroke` provider.
- [x] **3.3 Commit stroke on pointer-up.** On pointer-up, finalize the in-progress stroke into the active layer via `addElementToActiveLayer`; clear `currentStroke`. Committed elements render too (naive full repaint is fine for now). Widget test: simulate a drag, assert the active layer gained a `Stroke` with >1 points.
- [x] **3.4 Input thinning.** Drop incoming points closer than a minimum distance (e.g. 1.5px in document space) to the previous point. Unit tests on the thinning function.
- [x] **3.5 Stroke smoothing.** Pure-Dart incremental quadratic-midpoint smoothing in `engine/smoothing.dart`, applied to the in-progress stroke as points stream in. Chosen over Catmull-Rom because every emitted point is a convex combination of its inputs, so the curve cannot overshoot the input bounds. Unit tests: smoothed output has expected point counts and stays within bounds of input.

## Phase 4 — Brush Feel

- [x] **4.1 Pressure → width.** Render each stroke as a filled variable-width path: width = `baseWidth * lerp(minFactor, 1.0, pressure)`. Unit test the width function; visual check for the path.
- [x] **4.2 Brush settings state.** `Brush {colorRGBA, baseWidth}` provider; new strokes read from it. Simple toolbar controls: width slider (1–64) and 8 hardcoded color swatches.
- [x] **4.3 Color picker dialog.** Tapping a "custom color" swatch opens an HSV picker dialog; picked color becomes the brush color and is added to a recent-colors row (max 8, most recent first). The same picker is reused for shape stroke/fill in 9.7.
- [x] **4.4 Eraser tool.** Tool enum gains `{pen, eraser}` + toolbar toggle. Eraser strokes have `toolId=1` and render with `BlendMode.clear` within their layer. Rendering per layer must use `saveLayer` so erasing only affects that layer. Note: the eraser clears the layer's raster, so it erases shapes in that layer too — it is not a shape-delete.
- [x] **4.5 Stroke stabilization option.** Optional pull-string stabilizer with strength 0–10, off by default, selectable in toolbar. The pen drags an anchor on a taut string; movement inside the radius is slack, which is what kills tremor. Runs before smoothing: it decides *which* points exist, the smoother interpolates between them. Unit tests on the stabilizer function.

## Phase 5 — Performance Pass

- [x] **5.1 Layer caching.** Composite each layer's committed elements into a cached `ui.Image`; invalidate a layer's cache only when its elements change (keyed on the identity of `Layer.elements`, which `copyWith` preserves, so a rename or opacity nudge is a cache hit). Layer opacity is applied on composite, never baked into the image. Committing a stroke *appends*, so the cache extends the existing image with the new element rather than re-rasterizing the layer — a commit into a 500-stroke layer costs ~6ms instead of ~860ms. Undo, reorder and delete fall back to a full render.
- [x] **5.2 Incremental in-progress painting.** The layer stack is split into three `RepaintBoundary`s — below the active layer, the active layer, above it — and only the active one carries the live stroke, so only it repaints while drawing. The live stroke still composites *inside* its layer, which is what keeps the eraser correct. A pointer event costs one image blit plus one path (~9µs with 500 strokes committed) instead of re-rasterizing the document. Verified by paint counters in a widget test, and by `debugRepaintRainbowEnabled`.

## Phase 6 — Undo / Redo

- [x] **6.1 Command infrastructure.** `Command` interface (`apply(doc)`, `revert(doc)`), `CommandStack` with undo/redo stacks (cap 200). **One stack per `DocumentSession`**, not one globally — undo in one tab must never touch another. The session also exposes a `dirty` flag derived from whether any command has run since the last save. Unit tests, including that two sessions' stacks are independent.
- [x] **6.2 AddElementCommand.** Route stroke commit (pen and eraser) through it; Phase 9 reuses the same command for shapes. Tests: undo removes the element, redo restores it, new action clears redo stack.
- [x] **6.3 Undo/redo UI.** Edit-menu items + Ctrl/Cmd+Z, Ctrl/Cmd+Shift+Z (and Ctrl+Y on Windows) via `Shortcuts`/`Actions`. Buttons disable when stacks are empty.

## Phase 7 — Layers

- [x] **7.1 Layer panel UI.** Right-side panel listing layers top-to-bottom, showing name, visibility eye toggle, and active-layer highlight. Clicking selects the active layer.
- [x] **7.2 Add/delete layer commands.** `AddLayerCommand`, `DeleteLayerCommand` (deleting the last layer is disallowed). Panel buttons + tests including undo behavior.
- [x] **7.3 Reorder layers.** Drag-to-reorder in the panel via `ReorderableListView`, as `ReorderLayerCommand`. Tests.
- [x] **7.4 Layer opacity + rename.** Opacity slider per layer, committed as one command on release rather than one per pixel of the drag, and rename via double-click on the layer's name. Both as commands. Tests.
- [x] **7.5 Layer thumbnails.** 48px live thumbnail per layer in the panel, drawn straight from the shared layer cache image. No throttle needed: a thumbnail is a scaled blit of an image that already exists, and that image only changes when the layer's elements do.

## Phase 8 — Responsive Canvas

The document is not a fixed page: **its canvas follows the window, and the drawing scales with it.** Resizing the window wider makes every stroke bigger, keeping the composition you drew.

Scaling is always **uniform**. A window whose aspect ratio differs from the document's would otherwise stretch circles into ellipses and shear rotated shapes. Widening the window therefore adds blank canvas on the long axis rather than distorting what is already drawn.

This changes the document, but a window-driven resize is deliberately **not** undoable: undo would shrink the canvas while the window stayed put, and the layout would immediately resize it back. `ResizeCanvasCommand` exists and is undoable for resizes the user explicitly asks for.

- [x] **8.1 Element scaling.** `CanvasElement.scaled(factor, {originX, originY})` on the sealed type: `Stroke` scales its points and `baseWidth`, `Shape` scales `x`/`y`/`w`/`h` and `strokeWidth` while leaving `rotation` alone. Uniform factor only — a non-uniform one has no correct meaning for a rotated shape. Unit tests: bounds scale by the factor, pressure and rotation survive, a scale about a point leaves that point fixed, and `scaled(s)` then `scaled(1/s)` returns to within float tolerance. This is also the geometry task 9.5 needs for resize handles.
- [x] **8.2 ResizeCanvasCommand.** Captures the old canvas size *and the old layer list* — element references, so it is cheap — then `apply` scales every element by `min(newW/oldW, newH/oldH)` about the canvas centre and recentres the result. `revert` restores the captured layers verbatim rather than dividing by the factor, so undo is exact instead of accumulating float drift. `apply` recomputes from the captured layers, so redo is repeatable. Tests: undo is lossless, redo is idempotent, an empty document resizes, a degenerate size is refused.
- [x] **8.3 Canvas follows the window.** The active session's canvas size tracks the canvas viewport in logical pixels, clamped to a minimum (e.g. 320×200) so a collapsed window cannot destroy a drawing. The page is drawn at 1:1, so `CanvasView.fitScale` collapses to the identity until Phase 14 introduces zoom. Widget test: growing the window scales the strokes and leaves their relative positions intact.
- [x] **8.4 Coalesced resizes.** A window drag emits a resize per frame; debounce (~250ms after the last) and apply the resize once. **Window-driven resizes are not undoable.** Making them commands deadlocks: undo shrinks the document while the window stays put, the layout sees the mismatch and resizes it again, and undo fights the window forever. Like zoom, this is the view adapting — it just happens to rewrite document state. `ResizeCanvasCommand` remains, and *is* undoable, for a resize the user asks for (13.3's canvas-size dialog). The drag previews through the view transform, never by rewriting the document per frame. Unit tests on the debouncer; widget tests that a drag adds nothing to the undo stack and that an explicit resize does.
- [x] **8.5 Fit-to-window toggle + status bar.** A per-session preference, on by default. Off means the document keeps the canvas size it was created or opened with, and the page merely scales to fit the viewport — the behaviour of Phases 0–7, and what you want after opening a `.skd` at a fixed size. Status bar shows the canvas size and the on-screen scale.

## Phase 9 — Shapes & Selection

Shapes are parametric and stay editable forever: resizing a rectangle changes its `w`/`h`, it never resamples pixels. Selection, move, rotate, delete, duplicate and z-order operate on **any** `CanvasElement`; only resize/rotate behave differently per type (shapes adjust their parameters, strokes transform their point list).

- [x] **9.1 Shape tool + drag-to-create.** Tool enum grows to `{select, pen, eraser, rectangle, ellipse, line, arrow, diamond}` with toolbar buttons. Dragging on the canvas creates a shape from the drag rect; Shift constrains to square/circle/45° line, Alt draws from the center. Live preview while dragging; on pointer-up commit via `AddElementCommand`. Widget test: a drag with the rectangle tool adds exactly one `Shape` to the active layer.
- [x] **9.2 Shape rendering.** Pure function `(ShapeType, Rect) → ui.Path` for all five types (arrow = shaft + head sized relative to `strokeWidth`; diamond = 4-point polygon). Painter fills then strokes, honoring `strokeStyle` (solid/dashed/dotted) and `rotation`. Unit tests on path bounds and segment counts per type.
- [x] **9.3 Hit-testing + selection state.** Pure `hitTest(element, point, tolerance)`: shapes test their filled region when filled and their stroked outline otherwise, applying the inverse rotation first; strokes test distance to any segment. Phase 10 adds the text case, which tests the box. Selection is a `Set<String>` of element ids **stored on the `DocumentSession`**, so each tab remembers what it had selected. Select tool picks the topmost hit across visible layers; Shift-click toggles; dragging from empty canvas draws a rubber-band marquee. Unit tests on the hit-test math, including rotated shapes.
- [x] **9.4 Selection overlay + move.** Draw the selection bounding box, 8 resize handles, and a rotation handle above the box — all at constant *screen* size. Dragging inside the selection moves it; arrow keys nudge 1px, Shift+arrow 10px. `MoveElementsCommand` coalesces one drag gesture into one undo entry. Tests.
- [x] **9.5 Resize.** Drag any of the 8 handles, reusing the element scaling from 8.1. Shapes resize losslessly (`x`/`y`/`w`/`h`); strokes scale their points about the fixed opposite anchor. Shift preserves aspect ratio; Alt resizes about the center. Dragging a handle past its anchor must flip cleanly (negative `w`/`h` normalize). `ResizeElementsCommand`. Unit tests per handle, including both flip axes.
- [x] **9.6 Rotate.** The rotation handle rotates the selection about its center; Shift snaps to 15° increments. Shapes store `rotation`; strokes rotate their points. `RotateElementsCommand`. Tests, including that 9.3 hit-testing agrees with the rendered result.
- [x] **9.7 Shape style controls.** Toolbar controls for stroke color, fill color (including "none"), stroke width, and stroke style. They set the defaults for new shapes and, when a selection exists, apply to it via `StyleElementsCommand`. Reuses the 4.3 color picker. Tests.
- [x] **9.8 Delete, duplicate, z-order.** Delete/Backspace removes the selection; Ctrl/Cmd+D duplicates it offset by (10, 10) and selects the copy; bring-forward / send-backward / bring-to-front / send-to-back reorder within the owning layer. All are commands. Tests including undo of each.

## Phase 10 — Text

A third `CanvasElement`: a **text box** with rich runs. It carries a font family, a maximum font size, and a colour; within it, any range of characters can be **bold, italic, or underlined**.

Two decisions shape the phase:

**Text uses system fonts, and the `.skd` stores the family name.** A file opened where that family is missing falls back to the platform default, so the *glyphs and line breaks change*. Rendering is therefore not byte-identical across machines: format tests must assert layout invariants and round-trip equality, never rendered pixels, and the golden fixture (11.5) asserts that a text element **loads**, not that it paints the same. Bundled fonts, which would make rendering deterministic, are in Later / Ideas.

**A text box has a fixed size, and the text shrinks to fit it.** The stored font size is a *maximum*; the renderer picks the largest scale ≤ 1 at which the wrapped text fits the box. Nothing is ever clipped or hidden — but the size you chose is not always the size you get. The fit scale is derived, never stored, so it can never disagree with the text. Below a floor (25%) it stops shrinking and the box shows an overflow marker rather than becoming unreadable.

Adding a variant to the sealed `CanvasElement` makes every non-exhaustive `switch` a compile error. That is the point: the codec, the renderer, and hit-testing cannot silently forget about text.

- [x] **10.1 TextElement + TextRun models.** `TextRun {text, bold, italic, underline}` and `TextElement extends CanvasElement {x, y, w, h, rotation, fontFamily, fontSize, colorRGBA, align, runs}`. The element's text is the concatenation of its runs, so ranges never overlap. Invariants: no empty runs, adjacent runs with equal styling are merged. `copyWith`, equality, rotation-aware `bounds` from the stored box — the domain stays Flutter-free and never lays text out. Unit tests, including that adding the variant breaks a non-exhaustive switch.
- [x] **10.2 Text layout + shrink-to-fit.** `engine/text_layout.dart`: build a `ui.Paragraph` from a `TextElement`, wrapped to `w`, and find the largest `fitScale ∈ [0.25, 1]` whose laid-out height fits `h`. Binary search to a fixed iteration count so the result is deterministic and cheap. Also exposes caret ⇄ offset mapping for 10.5. Unit tests: a narrower box wraps to more lines; a taller box needs no shrink; the scale is monotonic in box height; a box too small clamps at 0.25 and reports overflow.
- [x] **10.3 Text rendering.** Paint the paragraph inside the rotated box, honouring per-run bold/italic/underline, the fit scale, and the element colour. Replaces the `TextElement` case that 10.1 forced into the painters. An overflow marker when clamped. Layer-cache invalidation already keys on the element list, so an edit re-rasterizes its layer only.
- [x] **10.4 Text tool + create.** Tool `text` (`T`) in the toolbar. Click places a default-sized box; drag creates one of that size. Commits via `AddElementCommand` and drops straight into edit mode. Widget test: a click adds exactly one `TextElement` and focuses it.
- [x] **10.5 In-canvas editing.** Caret, character input, IME, backspace/delete, arrow keys, Home/End, click to place the caret, drag and Shift+arrow to select a range, Ctrl/Cmd+A to select all. Typing coalesces into one `EditTextCommand` per burst (idle ~500ms, or on blur), so undo steps back a word rather than a keystroke. Escape and clicking away commit.
- [x] **10.6 Resize and reflow.** A **corner** handle scales the box and `fontSize` together, so the text keeps its apparent size. A **side** handle emits `ResizeBoxCommand` instead: it changes one axis of the box and leaves `fontSize` alone, so `layoutText` rewraps and re-fits the text into the new width rather than magnifying it. `elementWithBox` is the split; `canResizeBox` restricts it to a lone unrotated boxed element — a rotated element's `bounds` is only its axis-aligned hull, and a stroke has no box — everything else falls back to the uniform scale. Undo restores the element verbatim. Tests cover the geometry, the fallbacks, and that a right-edge drag leaves `fontSize` at 20.
- [x] **10.7 Bold, italic, underline on a selection.** Toolbar buttons and Ctrl/Cmd+B / I / U. With a range selected they apply to it, splitting the runs it straddles; with nothing selected they apply to the whole element. Toggling twice restores the original run list exactly — the merge rule from 10.1 is what makes that true. `StyleTextRunsCommand`. Unit tests on run splitting and merging: styling the middle of a run yields three, un-styling it collapses back to one.
- [x] **10.8 Font family, size, and colour.** A size field, a family picker and a colour swatch (the 4.3 HSV dialog) in the toolbar; each restyles the selected box through `StyleTextElementCommand`, or sets the default for the next one. Flutter cannot enumerate or query installed fonts, so `system_fonts.dart` asks fontconfig (`fc-list`) on Linux and confirms every candidate with `isFontAvailable`, which *measures*: a family the engine cannot resolve lays out identically to one that certainly does not exist. When a file names a family this machine lacks, the picker still shows that name and a **"missing"** notice explains the fallback rather than letting the box look wrong for no reason. macOS enumeration lands with its build in Phase 16.

## Phase 11 — The .skd File Format

- [x] **11.1 Element binary codec.** `element_codec.dart`: encode/decode `List<CanvasElement>` ↔ bytes per the spec in CLAUDE.md (magic, count, per-element `elementType` discriminator, float32 LE). Strokes, shapes and text elements all round-trip; text carries its family name and run list as length-prefixed UTF-8. Unit tests: round-trip equality for strokes, every shape type, and text with mixed runs (including non-ASCII); bad magic rejected, truncated data rejected, unknown `elementType` rejected.
- [x] **11.2 Manifest + document JSON.** Serializers for `manifest.json` and `document.json` with `toJson`/`fromJson` and validation (unknown blend modes → "normal"; missing fields → error). Unit tests.
- [x] **11.3 SkdWriter.** Assemble ZIP: uncompressed `mimetype` first entry, manifest, document.json, one `elements/<id>.bin` per layer. Writes atomically (temp file + rename). Test: output unzips with standard tools and entries match spec.
- [x] **11.4 SkdReader.** Open ZIP → validate mimetype + formatVersion → parse manifest/document → decode element blobs → return `SkdDocument`. Errors are typed (`SkdFormatException` with reason). Tests: full round-trip write→read→deep-equals, corrupt zip rejected, formatVersion 999 rejected with clear message.
- [x] **11.5 Golden fixture.** Commit a small hand-verified v1 `.skd` file to `test/fixtures/` containing at least one stroke, one of each shape type, and a text element with mixed bold/italic/underline runs; add a test that it always loads and deep-equals the expected model. It must **not** assert rendered pixels: text uses system fonts, so glyphs and line breaks differ across machines. This locks backward compatibility forever.
- [x] **11.6 Thumbnail generation.** Render the flattened document to a max-256px PNG and include as `thumbnail.png` in the ZIP. Test: entry exists and decodes as a valid PNG with correct aspect ratio — not that its pixels match a golden, since text renders with whatever fonts the machine has.

## Phase 12 — Multi-Document Tabs

The state has been per-session since 2.5; this phase gives it a face. It lands before the File menu so that New, Open, Save and quit are written tab-aware from the start rather than retrofitted afterwards.

- [x] **12.1 Tab bar UI.** `TabStrip` between the menu bar and the canvas: one tab per `DocumentSession`, its title, a dot when dirty, and a close button. Clicking a tab calls `setActiveSession`; `+` opens a blank one. Hidden entirely when a single document is open — a lone tab is a title bar that steals 34px and says nothing. `Untitled N` is fixed when the session opens, not derived from its position, so closing tab 1 does not renumber tab 2 under the pointer.
- [x] **12.2 Tab lifecycle + shortcuts.** Ctrl/Cmd+T opens, Ctrl/Cmd+W closes, Ctrl+Tab and Ctrl+Shift+Tab cycle with wrap, Ctrl/Cmd+1..8 jump to the nth and Ctrl+9 to the *last* (as everywhere else). An out-of-range jump does nothing rather than throwing. Closing a dirty tab brings it to the front and prompts Save / Don't save / Cancel — dismissing the dialog counts as Cancel, since it is not consent to lose the document. Closing the active tab focuses the one that slid into its place; closing the last leaves one fresh empty session.
- [x] **12.3 Tab reorder + overflow.** Drag to reorder (`moveSession`, keeping the same document in front). Tabs share the strip until they would be narrower than `kTabMinWidth`, past which the strip scrolls rather than shrinking them to slivers. Middle-click closes.

## Phase 13 — File Menu Integration

- [x] **13.1 Save / Save As.** File menu + Ctrl/Cmd+S (Shift for Save As), acting on the active session, through a `FileService` seam — `file_selector` and the real disk in production, an in-memory fake in tests, so no widget test opens a GTK chooser. Plain Save overwrites silently; a failed write says so and leaves the document dirty. A document edited *while* the bytes were being written is named but stays dirty. Tab titles switch to the file's base name, and the window title becomes `<file> — InkPad` with a `*` when unsaved. Flutter exposes no Dart API for the desktop window title, so `linux/runner/my_application.cc` gained an `inkpad/window` method channel; a platform with no handler (macOS until Phase 16, and every test) is a no-op, not an error.
- [x] **13.2 Open.** Loads a `.skd` into a new tab, never replacing the current one, and keeps the canvas size stored in the file, so fit-to-window starts off for it. A path already open focuses its existing tab rather than opening a second view that would silently overwrite the first on save. Corrupt and unsupported files show the reason from `SkdFormatException` and open no tab.
- [x] **13.3 New + dirty tracking.** File → New opens a tab after a canvas-size dialog (presets, custom, clamped to `kMinCanvasWidth`..`kMaxCanvasWidth`, plus a fit-to-window choice that leaves 8.5 enabled and is on by default). Dirty state is per session; the tab shows a dot and the window title a `*`. Quitting reviews each dirty session in turn — a single "discard everything" button would be one misclick from losing three tabs of work — and cancelling any one of them cancels the quit.
- [x] **13.4 Recent files.** File → Open Recent (last 8), persisted with `shared_preferences`, newest first, no duplicates. Missing files are pruned when the list loads, and a file that fails to open is forgotten rather than offered again. `add`/`remove` await the initial load before writing: a naive `state.value` write is undone the moment the prune finishes.
- [x] **13.5 Autosave.** Every 3 minutes each dirty session writes a sidecar — `<file>.skd.autosave` beside a saved document, or a scratch file keyed by *session id* for an `Untitled` one, since titles collide across runs. One session's write failing does not stop the others. On open, a sidecar newer than the file itself offers recovery; recovered work keeps the real file's path but counts as dirty, because it has never been written there (`CommandStack.markUnsaved`). A successful manual save, or closing the tab, deletes the sidecar. The scratch directory is injected rather than resolved directly: `path_provider`'s channel never answers under `flutter_test`, so awaiting it hangs instead of failing.
- [x] **13.6 OS file association.** `packaging/linux/inkpad-skd.xml` registers `application/x-skd` with a glob and a magic rule that matches the ZIP header plus the stored `mimetype` entry the format spec requires first (verified against the golden fixture: `mimetype` at offset 30, the type string at 38) — the same trick ODF and EPUB use, and the reason `mimetype` is stored rather than deflated. The `.desktop` file gains `MimeType=`, the AppDir ships the definition, and `main(List<String> args)` opens any `.skd` in `argv`. The AppImage smoke test now runs twice: bare, and with the golden fixture as `argv[1]`. Windows and macOS registration land with their builds (Phase 16); the app has no Windows target and no macOS build yet.

## Phase 14 — Canvas Navigation

- [x] **14.1 Zoom.** Ctrl+scroll zooms about the cursor (10%–800%, clamped not refused); Ctrl+= / Ctrl+- zoom about the viewport centre; Ctrl+0 resets to fit, as does clicking the zoom in the status bar. `ViewTransform` lives on the `DocumentSession`, so each tab restores its own zoom and scroll, and it is never a command: zooming to 800% and back leaves the document and the undo stack untouched. `zoomAround` solves for the pan rather than iterating, so the pixel under the cursor stays put and zooming in and back out returns to exactly where it started. Drawing and hit-testing are tested at 10% and 800% — note the smallest canvas is 320×200, so at 8× the page always overflows the viewport and a test must aim at a visible point.
- [x] **14.2 Pan.** Space-drag and middle-drag pan; the cursor becomes a hand the moment space goes down, and a grabbing hand while dragging. `StrokeCapture` ignores any non-primary button and every drag while space is held, so a pan never leaves a stroke behind. Panning does not clear `fitted`: dragging a page aside says nothing about how big it should be.
- [x] **14.3 Pinch/trackpad gestures.** `PointerPanZoomStart/Update` events: two fingers pan, a pinch zooms about the gesture's position, applied as a ratio against the previous update rather than an absolute. A plain mouse wheel pans, and pans horizontally with its horizontal delta.

## Phase 15 — Export & Polish

- [x] **15.1 Export PNG.** File → Export PNG: 1×/2×/4× with an optional transparent background, through the same `DocumentPainter` the canvas uses — so what is exported is what was on screen, honouring layer visibility, opacity and blend modes. A scale whose surface would exceed `kMaxExportEdge` is offered disabled and refused with a reason rather than dying. Exporting is not saving: it never gives the session a `filePath`, never marks it clean, and never touches Open Recent. Note for future tests: `Picture.toImage` needs the real event loop, so a widget test must let it run under `tester.runAsync`.
- [x] **15.2 Keyboard shortcuts reference.** V/B/E/R/O/L/A/D pick tools (plus T for text), `[` and `]` step the brush width and stop at the ends, and Help → Keyboard Shortcuts (or F1) lists everything. The single-letter bindings are suppressed while a text box is being edited — the shell hands the keyboard to the box — or typing "bold" would swap tools four times and never reach it.
- [x] **15.3 Preferences window.** Edit → Preferences, persisted with `shared_preferences`: default canvas size and fit-to-window for File → New, autosave interval (0 turns it off, and the ticker restarts when the interval changes), default brush, default shape style, and theme (light/dark/system). Nothing is written until Save, so a slider dragged then cancelled leaves the stored settings alone. A stored value out of range is clamped and an enum index no build has falls back, so a hand-edited file cannot produce a 0px brush.
- [x] **15.4 Crash-safe format stress test.** Property-based round-trips over fixed seeds (random layers, all element types, all five shape types, non-ASCII and emoji text, a 5000-point stroke), plus reader fuzzing: single-bit flips, truncation at every offset, random noise, and corrupted element blobs. **This found a real bug**: `archive` inflates lazily, so a bit-flip threw `FormatException: Filter error, bad data` out of `file.content` — escaping the reader's contract that it throws only `SkdFormatException`. Two format facts are now pinned by name rather than discovered by surprise: `rotation` is a float32 (0.97 reads back 0.9700000286), and `document.json` stores the background as `#RRGGBB`, so its alpha byte lives only in memory.
- [x] **15.5 App icon + naming.** The name is **InkPad**. `images/inkpad.ico` is the Windows runner icon and the macOS `AppIcon.appiconset` is regenerated from `inkpad_icon_1024.png`; Linux already shipped its hicolor icons from 1.2. The GTK window icon is set by handing the runner the shipped PNG's **bytes** over the `inkpad/window` channel, not by `gtk_window_set_icon_name`: an AppImage cannot resolve a name against the host's icon theme, which is what produced the "Could not load a pixbuf from icon theme" warning. Decoding pixels we ship skips the lookup. Window title is `<file> — InkPad`, starred when unsaved (13.1). *Not verified on a real desktop* — the handler compiles and the AppImage smoke test still starts; that the icon appears in a taskbar has not been seen.

## Phase 17 — Rough Rendering, Snapping, Groups & Connectors

Four features from the ideas list. Two of them add `CanvasElement` variants, which the sealed type turns into compile errors until the codec, the renderer and hit-testing all handle them. That makes this the first **format v2**: v2 readers still open every v1 file, and a v1 reader refuses a v2 file with "written by a newer version" rather than choking on an unknown `elementType`.

Connectors cannot store the id of what they bind to — `.skd` regenerates element ids on load, and nothing in the format may reference one. They bind by **index within their container** instead, which is stable because element order *is* z-order.

- [x] **17.1 Rough rendering.** `Shape` gained `renderStyle` and a real `seed`. The jitter is a pure function of `(seed, point index)` — an integer hash, not a sequential PRNG, so resizing a shape or changing the segment count cannot reshuffle it. Two passes per outline; the fill always follows the *precise* path, because a rough outline is a handful of open subpaths and filling that paints nonsense. Hit-testing keeps using the parametric outline. A shape toggled to rough after creation takes a seed derived from its id (`seedFromId`), not a random one: `transform` runs again on redo, and a random seed there would wobble differently from the apply it is meant to reproduce.
- [x] **17.2 Snapping and alignment guides.** `engine/snapping.dart` is pure: moving box + target boxes + threshold + optional grid → chosen delta and the guides to draw. Each axis is decided independently by the nearest candidate, so a box can snap its left edge to one element and its top to another. The threshold is in *screen* pixels, divided by the page scale, so a snap feels the same at 10% and at 800% zoom. Elements beat the grid at equal distance; a grid snap draws no guide, because the grid is already visible. Alt suspends snapping. Toggles live in the status bar.
- [x] **17.3 Group / ungroup.** `Group` holds an ordered, nestable child list. Ctrl+G groups into the topmost member's z-position and keeps the children's relative order; Ctrl+Shift+G ungroups one level, leaving inner groups intact. A group transforms as one rigid body — it has no rotation field of its own, so the children carry the angle — and hit-tests as one element. Both commands capture the layer's element list and restore it verbatim, so undo is exact. Duplicating a group freshens every nested id and rebinds the copied connectors onto the copied shapes (`withFreshIdsAll`); reusing an id would leave the document with two elements answering to the same click.
- [x] **17.4 Connectors.** A `Connector` joins two endpoints, each a free point or bound to a sibling. **A bound end stores no coordinates**: it is the point where the line between the two anchors leaves the bound element's box, resolved on every draw. So moving, resizing or rotating a shape needs no command that touches the connector and undo stays exact — the fact the tests pin is that a connector's *object* never changes when the shape it follows moves. Deleting a bound element freezes that end where it stood, and undo rebinds it (the before/after pair is captured, not recomputed). Grouping freezes any end that reaches outside the new group. Persisted as an index into the container's element list, since ids are not in the format.

## Phase 19 — Brush Engines, Rich Text & Infinite Canvas

Four features from the ideas list. Two need format changes, decided up front: per-run font size and colour bump to **v3**, and infinite canvas is a **per-document mode** (v3 as well, since it lands in the same push). Brush engines need no bump — the `toolId` byte was never validated on read, so a new id degrades to the pen in an older build. Text-on-a-path **binds to a sibling element**, the way a connector does (17.4), so it follows what it is attached to.

- [x] **19.1 More brush engines.** `ToolId` gains `pencil` (2), `airbrush` (3) and `texture` (4). Rendering dispatches on `toolId` in `paintStroke`: pencil is a translucent fill with seeded graphite grain, airbrush a soft blurred low-opacity spray, texture a row of broken dabs stamped along the resampled centreline. The grain and dabs are seeded from the stroke's own **points** (`brushSeed`), not a stored field, so they survive save/reload with no format change and no two strokes share a pattern. An unknown `toolId` from a newer file renders as a pen rather than being rejected. Toolbar buttons for each; the geometry (`buildTextureStamps`, `buildPencilGrain`, `resampleByArcLength`) is pure and tested for determinism.
- [x] **19.2 Per-run font size and colour.** `TextRun` gained an optional `fontSize` and `colorRGBA` (null = inherit the element's), marked by `styleFlags` bits 3 and 4 so the codec writes the extra `f32`/`u32` only when set — self-describing, so a v3 reader parses v1/v2 runs unchanged. **Format is now v3.** `restyleRange` splits a straddled run and `runsWithFontSize`/`runsWithColor` set or clear the override; runs merge only when size *and* colour match, so styling stays reversible. Layout applies the per-run size (riding the shrink-to-fit scale) and colour. The toolbar size field and colour button apply to the selected range when a box is being edited, else to the element's own defaults. `TextElement.scaled` scales per-run sizes too, or a corner-resize would leave overridden runs behind. Golden `v3_golden.skd`, plus stress-test coverage.
- [x] **19.3 Bulleted/numbered lists and text on a path.** A per-element `ListStyle` (none/bullet/numbered) carried in a text-body reserved byte, so v1/v2/v3 files read as plain — no version bump. Layout prefixes each line with its marker (`listMarker`). And a text element can **bind to a sibling** (`pathElementId`, stored as an index behind a self-describing flag byte, resolved two-pass like a connector), flowing its glyphs along that element's outline (`paintTextOnPath` walks the path by arc length, rotating each glyph to the tangent) and following it when it moves. Hit-testing and bounds follow the outline. Toolbar buttons for both; `toggleTextOnPath` binds one text plus one outlined sibling, or detaches. Known rough edge: the selection overlay still shows the text's own box, not the glyphs on the path.
- [x] **19.4 Infinite canvas mode.** A per-document `CanvasMode` (bounded/infinite), stored as `canvas.mode` in `document.json` — omitted when bounded, so older files and re-saved bounded documents are byte-identical, and a missing/unknown value reads as bounded (no forced version bump; it rides v3). An infinite document ignores the stored size, never resizes to the window, drops the white page, and pans/zooms over an unbounded plane. Because the layer cache rasterises each layer to a fixed `width × height` image — which cannot hold unbounded, possibly-negative content — infinite documents bypass the cache and repaint live through `InfiniteCanvasPainter`. Input, selection, guides, grid and the text editor take a document-origin offset so the same overlays serve both modes; export and thumbnails derive their region from the content. New documents can be created infinite from the New dialog. Known trade-off: live repaint is slower than the cached bounded path.

## Phase 18 — Shortcut & Navigation Gaps

Small refinements to input handling, mostly filling gaps in the shortcut map. Some of what was asked for already ships and needs no work: **Ctrl/Cmd+S** saves (13.1), **Ctrl/Cmd+W** closes the active tab (12.2), and **Ctrl/Cmd+= / Ctrl/Cmd+-** zoom about the viewport centre (14.1). The tasks below are the parts that are genuinely missing.

Modifier keys follow the existing convention: bind Control and Meta both rather than switching on the platform, so an Apple keyboard on Linux and the widget tests work either way.

- [ ] **18.1 Middle-mouse scroll to zoom.** Turning the wheel while the middle button is held zooms about the cursor, the same zoom `Ctrl+scroll` already performs (14.1). Today the middle button only pans by drag (14.2) and a plain wheel pans; this adds a distinct wheel-to-zoom gesture for mice without a scroll modifier within reach. The pan-by-middle-drag and Alt-suspends-snapping behaviours stay as they are, so the middle button now does two things depending on whether the wheel turns — verify the drag-pan is not triggered by the zoom, and vice versa. Pure logic sits in the `_NavigationLayer`'s `onPointerSignal`, gated on `HardwareKeyboard`/pointer button state; widget test that a wheel event with the middle button down changes the session's `ViewTransform.zoom` and holds the cursor point, and that it never touches the document.
- [ ] **18.2 Export shortcut.** Ctrl/Cmd+E opens the Export PNG dialog (15.1), which today is reachable only from File → Export PNG. A new `ExportIntent`, an activator pair in `kAppShortcuts`, a `CallbackAction` in `AppShell`, and the `shortcut:` annotation on the menu item so it shows the accelerator. Suppressed while a text box is being edited, like every other single-key-free binding (the shell already drops `kAppShortcuts` when `textEditingProvider` is non-null, so this is automatic once the intent lives there). Add the row to the Help → Shortcuts reference (15.2). Widget test: the shortcut opens the dialog, and firing it while typing in a text box does not.
- [ ] **18.3 Quit — shortcut and menu item.** Ctrl/Cmd+Q, plus a **File → Quit** item at the bottom of the menu with the accelerator shown. Both route through the existing `confirmQuit` (13.3), which reviews every dirty session before closing and returns whether quitting may proceed — so the shortcut cannot silently discard unsaved work any more than closing the window can. On success it calls `SystemNavigator.pop()`, exactly as the window-close path in `inkpad_app.dart` already does; factor that into one "attempt to quit" helper rather than duplicating the flow. Add the row to the Help → Shortcuts reference. Widget test: with a dirty tab open, the shortcut shows the save prompt and Cancel leaves the app running; with everything clean it proceeds. `SystemNavigator.pop` is a platform channel with no host under `flutter_test`, so the test asserts up to the decision, and stubs or tolerates the final call.
